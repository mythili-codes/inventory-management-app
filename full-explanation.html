📦 INVENTORY MANAGEMENT APP Full-stack web application ════════════════════════ 💡 WHY I BUILT THIS Small business owners — local shop owners, home business people, Instagram sellers, boutique owners — struggle every day to track their inventory. Right now they use: • Notebooks (get lost, damaged, hard to share) • Excel sheets (messy, confusing, not real-time) • Memory (forget, mistakes) Big companies use enterprise ERP systems like SAP that cost lakhs. Small businesses can't afford this. I wanted to build a simple, free web app that gives small business owners the same power — clean stock tracking, low-stock alerts, real-time value — accessible from any phone or laptop. ════════════════════════ 🎯 THE REAL PROBLEM I SOLVED The hardest technical challenge in multi-user apps is data isolation. When many users share the same app, how do you make sure User A never sees User B's data? Most beginner developers write permission checks in every part of the code. The problem: if you forget one check, you leak data. That's a serious security issue. I solved this using Row-Level Security (RLS) in PostgreSQL. Every database query is automatically filtered based on the logged-in user — at the database level, not in application code. Even if my frontend had bugs, the database would still refuse to leak data. This is how production apps like Notion, Airtable, and Linear handle security. I learned to think like a real backend engineer. ════════════════════════ 👥 WHO USES THIS • Local grocery / provision shop owners • Instagram / WhatsApp online sellers (dresses, cosmetics, snacks) • Home bakers, boutique owners, home business • Small medical stores, stationery shops, gift shops • Any freelancer tracking inventory • Anyone who currently uses notebooks/Excel to track stock ════════════════════════ ✨ HOW IT HELPS USERS BEFORE (with notebooks): ❌ Time-consuming manual entry ❌ Easy to lose or damage ❌ Can't check from anywhere ❌ Manual calculations for total value ❌ Manual scanning for low-stock items ❌ Shared access is hard AFTER (with my app): ✅ Add items in 10 seconds ✅ Real-time stats — total items, units, value ✅ Automatic low-stock alerts (red highlights) ✅ Access from any phone or laptop ✅ Each user's data is private and secure ✅ Free forever, no credit card ════════════════════════ 🧱 TECHNICAL STACK Frontend: • HTML5 (semantic markup) • CSS3 (design system with tokens) • Vanilla JavaScript (no framework) • Feather-style SVG icons Backend + Database: • Supabase (Backend-as-a-Service) • PostgreSQL relational database • Row-Level Security policies • JWT-based authentication Deployment: • Git + GitHub (version control) • GitHub Pages (static hosting) • Automatic CI/CD on git push Design: • Custom design system with color tokens, typography scale, spacing grid • Reusable components (buttons, inputs, cards, tables, badges, toasts) • Fully responsive: desktop, tablet, mobile • WCAG accessibility (contrast, ARIA labels, keyboard navigation) ════════════════════════ 🧠 KEY TECHNICAL DECISIONS 1. Vanilla JavaScript (no React) Reason: I wanted to understand what frameworks abstract — direct DOM manipulation, event handling, state management. Once I understand these, React becomes a tool I choose consciously, not a mystery I depend on. 2. Supabase over custom backend Reason: Building a backend from scratch takes weeks. I wanted to focus on what makes apps actually secure (Row-Level Security policies) rather than boilerplate authentication code. Supabase gave me production-grade auth and Postgres out of the box. 3. Row-Level Security instead of code-level checks Reason: Code checks fail if a developer forgets one place. Database checks never forget. Data isolation should be enforced at the deepest layer possible — the database itself. This makes security impossible to bypass. 4. GitHub Pages hosting Reason: Netlify hit free tier limits mid-project. I migrated to GitHub Pages within an hour — unlimited free hosting for static apps. Real-world lesson in choosing hosting infrastructure. 5. Design system from scratch Reason: My first version had emojis and gradients — looked AI-generated. Real users flagged this. I built a proper design system with color tokens, typography scale, spacing grid, and reusable components. Now every page feels consistent, like a real SaaS product. ════════════════════════ 🎓 WHAT I LEARNED Real user feedback beats my own opinion I shipped early and let real users test it. Mouna caught a missing password toggle. Jeevith K flagged missing success feedback, AI-looking gradients, and a subtle UI bug. Ammu found that email confirmation was broken. Each round of feedback made the product measurably better. My taste alone would never have caught them. Silent failures are the hardest bugs Users reported "signup doesn't work" — clicking the button did nothing. Turned out Brave browser's ad blocker was silently blocking my CDN-loaded Supabase SDK. My JavaScript never ran, so the browser fell back to native form submission (page refresh with empty fields). Fix: self-host the SDK from the same domain. Lesson: when something "does nothing," look for silent JavaScript failures. Design systems make apps look real The difference between a portfolio project and a real product is consistency. Same padding scale (4, 8, 12, 16...). Same border-radius. Same button structure everywhere. Boring? Yes. Professional? Also yes. Shipping matters more than perfection Perfect never ships. I shipped the app rough, learned from real feedback, and iterated. Every iteration taught me something the tutorial didn't. ════════════════════════ 🔧 HOW THE AUTH FLOW WORKS 1. User signs up → Supabase creates account, returns JWT token 2. JWT stored in browser localStorage (managed by Supabase SDK) 3. Every database query attaches the JWT as an Authorization header 4. PostgreSQL validates JWT → extracts user_id → applies RLS policies 5. Only rows where user_id matches are returned 6. User sees only their own data — always, guaranteed ════════════════════════ 🌐 LINKS Live Demo: https://mythili-codes.github.io/inventory-management-app/ Source Code: https://github.com/mythili-codes/inventory-management-app LinkedIn: https://www.linkedin.com/in/mythili-s-bb3257404 ════════════════════════ 💬 WHAT'S NEXT Planned improvements: • Password reset flow • CSV bulk import for existing inventory • Categories and tags for items • Charts and analytics • Email notifications for low stock • Export to CSV/PDF for reporting • End-to-end tests with Playwright ════════════════════════ I'd love your feedback!
