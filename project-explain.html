1. 30-Second Pitch Quick intro

2. 2-Minute Pitch Recommended ⭐

For interviews, recruiters, network calls
I built a full-stack Inventory Management application — a web dashboard where any user can sign up, manage their own inventory items, and track stock in real time. The problem I wanted to solve was multi-tenant data security: when many users share the same app, how do you make sure User A never sees User B's data? Traditional approaches require writing authorization checks in every backend endpoint, and one missing check can leak data. Instead of that, I used PostgreSQL Row-Level Security policies at the database layer. Each RLS policy says "return this row only if user_id equals the current authenticated user." This means data isolation is enforced at the database — the frontend can't accidentally leak someone else's data even if I wrote a buggy query. The tech stack is intentionally simple: HTML, CSS, vanilla JavaScript on the frontend — no framework — and Supabase for authentication and the Postgres database. Auth uses JWT tokens stored in the browser. Deployment is GitHub Pages with automatic deploys on every git push. Along the way, I built a full design system with color tokens, a typography scale, spacing grid, and reusable components — buttons, inputs, cards, tables, badges, toasts — all with proper accessibility, focus states, and mobile responsiveness. What I learned most: real user feedback matters more than my own opinion. I shipped early, got feedback from multiple LinkedIn users — Mouna, Jeevith, Ammu, and others — and iterated with each round: password toggle, success feedback, form reset, redesign, black-spot bug fix. Each fix taught me something new. Live demo: mythili-codes.github.io/inventory-management-app Code: github.com/mythili-codes/inventory-management-app

3. Technical Deep Dive For engineers

When talking to senior developers, architects, tech interviewers
Architecturally, this is a static single-page application talking directly to Supabase — no custom backend. Frontend: vanilla JS with three main routes — signup, login, and dashboard. State is managed by the Supabase SDK, which handles JWT session persistence in localStorage. Authentication flow: signup calls auth.signUp, which either returns a session immediately (if email confirmation is off) or requires email verification. On the dashboard, I check for a session on load and redirect to login if none exists. Data model: a single 'items' table with user_id, sku, name, quantity, reorder_level, unit_price. The critical piece is the RLS policies — I've enabled RLS on the table and written policies for SELECT, INSERT, UPDATE, DELETE that all check user_id = auth.uid(). This means even if the frontend sent a query for someone else's data, Postgres would filter it out at the database level. Design system: I built this from scratch after user feedback. Design tokens for color, typography, spacing, radius, shadows — all as CSS custom properties. Components are pure HTML+CSS classes: btn, input, card, alert, badge, toast, table. They're consistent across every page. Accessibility: semantic HTML, ARIA labels on icon buttons, aria-live regions for toasts, focus-visible outlines, WCAG-compliant contrast ratios, keyboard-navigable interfaces. Deployment pipeline: GitHub Pages with automatic build on push. I actually migrated from Netlify to GitHub Pages mid-project when Netlify's free tier credits ran out — good lesson in choosing hosting infrastructure. The biggest technical challenge was a real production incident where Brave browser's ad blocker was blocking Supabase's CDN. Users would fill the form, click submit, and nothing would happen — the form would just refresh. I diagnosed it by looking at the network tab, fixed it by self-hosting the Supabase SDK from the same origin as the app. The whole codebase is ~2000 lines and covers signup, login, dashboard, and design system.
💼 Common Interview Questions
Q: Why did you build this project?
I wanted to learn how real production authentication works — not just tutorials, but actually shipping something. Inventory was a small enough domain to build end-to-end while still teaching me about auth, database security, deployment, and real user feedback.
Q: Why Supabase instead of building your own backend?
Supabase gave me production-grade Postgres, auth, and Row-Level Security out of the box. I wanted to focus on what makes web apps actually secure — RLS policies at the database — not reinventing authentication middleware. It let me learn what a real backend cares about without spending weeks building one.
Q: What's Row-Level Security and why does it matter?
RLS is a Postgres feature where policies at the database level filter rows based on the current user. My policy says "return this item only if user_id equals the authenticated user's id." This means even if my frontend has a bug that queries the wrong data, Postgres will still filter it — data isolation is enforced at the deepest layer. In traditional apps, you'd write this check in every API endpoint; one missing check leaks data. RLS makes it impossible to forget.
Q: What was your hardest bug?
Users reported signup wasn't working in Brave browser — they'd fill the form, click submit, and the page would just refresh with empty fields. It looked random. The root cause: Brave's ad blocker was silently blocking cdn.jsdelivr.net, so the Supabase SDK never loaded. My JavaScript threw an error before preventDefault ran, so the browser did a native form submit — the page refresh explained the empty form. Fix: I self-hosted the Supabase SDK from my own domain. Same-origin means no ad blocker can touch it. Lesson: silent failures are the hardest to debug. If a click "does nothing," the code either didn't run or the browser fell back to default behavior.
Q: What did user feedback teach you?
Ship early, ship rough, listen carefully. My first version had emojis in the UI, AI-looking gradients, missing success messages, no form reset after signup, and a subtle black spot on the login page. None of that felt wrong to me until real users told me. Mouna caught the missing password toggle. Jeevith flagged the missing success feedback, the default value in Reorder At, and the AI gradient look. Someone else found the black spot bug. Each piece of feedback made the product measurably better. My taste alone would never have gotten it there.
Q: Why no framework like React?
Two reasons. First, I wanted to understand what frameworks abstract away — direct DOM manipulation, event handling, state — so I'd know when to reach for React vs. when vanilla is enough. Second, this app's complexity didn't justify a build pipeline or virtual DOM. Vanilla JS ships in kilobytes and works instantly. If the app grew to have complex state, forms, or many pages, I'd migrate to React.
Q: How does the auth flow work?
On signup, the Supabase SDK calls auth.signUp which creates a user in Postgres. Because I turned off email confirmation, it returns a JWT session immediately. The SDK stores the JWT in localStorage. On subsequent visits, the SDK reads the JWT and attaches it to every database query. Postgres validates the JWT, extracts the user_id, and applies RLS policies. If the JWT is missing or expired, the query returns no rows. The dashboard checks for a session on load. No session, redirect to login. Login calls signInWithPassword which either returns a session or an error. Logout calls signOut which clears the JWT.
Q: What would you improve if you had more time?
A few things: 1) Password reset flow — currently missing. 2) Bulk import from CSV — users with existing inventory can't onboard fast. 3) Categories and tags — right now everything is flat. 4) Charts and analytics — visualize stock over time. 5) Export to CSV/PDF — for reporting. 6) Email notifications when items hit reorder level. 7) End-to-end tests with Playwright. I chose to stop at the current scope because shipping a polished, focused MVP taught me more than half-building a bigger one.

🎓 Key Learnings — For "What Did You Learn?"

Learning Story Use in interviews

Three big lessons from building this: 1. Real user feedback beats my own taste I shipped the app, and within days real users on LinkedIn tested it. They caught things I couldn't see: missing password toggle, missing success messages, AI-looking gradients, subtle UI bugs. My taste alone would never have caught them. The lesson: ship early, listen, iterate. 2. Silent failures are the hardest bugs Users reported "signup doesn't work" — the button clicked but nothing happened. It turned out Brave's ad blocker was silently blocking my CDN-loaded Supabase SDK, so my JavaScript never ran, and the browser fell back to native form submission — which just refreshed the page with empty fields. Diagnosing this took hours because there was no error message, no red text, nothing in the console visible to the user. The fix — self-hosting the SDK — took 10 minutes once I understood the problem. Lesson: when something "does nothing," ask what fell back to a default behavior. That's where the silent failure lives. 3. Design systems make apps look real My first version had emojis in headers, gradient backgrounds, inconsistent spacing. It looked like a tutorial project. After feedback, I rebuilt it with proper design tokens — one place to change color, spacing, typography — and consistent components. The same functionality, but suddenly it looked professional. Lesson: the difference between "portfolio piece" and "product" is often just consistency at the token level.
